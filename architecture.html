Entity Merge (Non-Lossy)
When stages 2/3 find a match, merger.js performs a non-lossy merge:

Merge Steps

  • Redirect all relations from duplicate → primary
  • Remove self-referencing relations
  • Deduplicate relation pairs (sum mention counts)
  • Migrate fact_entity links (INSERT ON CONFLICT)
  • Merge entity_types arrays
  • Mark duplicate with merged_with FK

Custom Entity Types

  • Types are arbitrary strings, not a fixed enum
  • Common: person, document, topic, service, team
  • Domain-specific: session, course, speaker, sprint
  • Entities can have multiple types (JSON array)
  • Relations: PART_OF, LED_BY, COVERS, AUTHORED_BY, ...
Reciprocal Rank Fusion (RRF) — Deep Dive
Merges vector (semantic) and keyword (lexical) search results into a single ranked list. Two parallel searches, one unified score.
score = weight / (K + rank + 1)

Constants

  • K = 20 — balances score spread for result sets of 5–50
  • VECTOR_WEIGHT = 1.0 — semantic search gets full weight
  • KEYWORD_WEIGHT = 0.7 — keyword search gets 70% weight

Why RRF?

  • Scores from different systems are incomparable (cosine vs ts_rank)
  • RRF only uses rank position, not raw scores
  • Items appearing in both lists get boosted naturally
  • Weights let us bias toward semantic (better for NL questions)

AUDM Deduplication — Deep Dive
Add / Update / Discard (skip) / Merge (contradict) — every new fact goes through this before storage. Prevents the knowledge base from growing stale or redundant.

Thresholds

  • SKIP ≥ 0.88 — essentially the same fact, discard the new one
  • AMBIGUOUS 0.65–0.88 — similar but not identical, ask LLM to decide
  • ADD < 0.65 — different enough to be a new fact

Tuned for nomic-embed-text: paraphrased content typically lands 0.75–0.88.

Embedding Model

  • Ollama nomic-embed-text (768 dimensions)
  • Cosine similarity via pgvector: 1 - (embedding <=> ?)
  • Top 5 similar facts retrieved per comparison
  • Only active facts considered (status != contradicted/superseded)

SELECT * FROM entity

Stage 2 — Fuzzy String Match (Levenshtein)

Algorithm

  • Fetch up to 200 candidates from namespace (not merged)
  • Compute Levenshtein distance for each
  • Similarity = 1 - (distance / max(len_a, len_b))
  • Filter: similarity ≥ 0.85 AND names not identical
  • Sort by similarity DESC, take top 3

What It Catches

  • "Kubernetes" ↔ "Kuberntes" (typo)
  • "JavaScript" ↔ "Java Script" (spacing)
  • "API Gateway" ↔ "Api Gateway" (case)
  • Does NOT catch: "K8s" ↔ "Kubernetes" (too different)
similarity = 1 − (levenshtein_distance / max(|a|, |b|))
Stage 3 — Embedding Similarity
// Embed format: "{entityType}: {name}"
// e.g. "service: API Gateway" → 768d vector

SELECT id, name, entity_type, entity_types,
       1 - (embedding <=> ?) AS similarity
FROM entity
WHERE namespace = ?
  AND embedding IS NOT NULL
  AND LOWER(name) != LOWER(?)
  AND merged_with IS NULL
  AND 1 - (embedding <=> ?) >= 0.85
ORDER BY embedding <=> ?
LIMIT 3
Catches semantic equivalents that fuzzy string matching misses: "K8s" ↔ "Kubernetes", "React.js" ↔ "ReactJS". Each candidate is verified by Claude Haiku before merging.

Graph Traversal — Deep Dive
Entities and relations form a directed graph. Traversal uses PostgreSQL recursive CTEs with cycle prevention and depth limiting.

Graph Structure

  • Nodes: entity table (person, document, topic, service, ...)
  • Edges: relation table (source_id → target_id)
  • Typed edges: PART_OF, AUTHORED_BY, COVERS, LED_BY, ...
  • Temporal: valid_at / invalid_at for time-bound relations
  • Filtered: merged entities excluded, only active relations

Defaults

  • findRelated: maxDepth = 2, limit = 30
  • findPath: maxDepth = 4, returns shortest
  • getEntityNeighborhood: depth = 1, limit = 50
  • Cycle prevention via path array check
  • DISTINCT ON (entity_id) keeps shallowest depth
findRelated() — Depth-Limited BFS
WITH RECURSIVE graph AS (
  -- Base case: direct relations from source
  SELECT r.target_id AS entity_id,
         r.relation_type, r.mention_count,
         1 AS depth,
         ARRAY[source_id] AS path
  FROM relation r
  WHERE r.source_id = ?
    AND r.invalid_at IS NULL

  UNION ALL

  -- Recursive: extend from current frontier
  SELECT r.target_id, r.relation_type,
         r.mention_count, g.depth + 1,
         g.path || r.target_id
  FROM relation r
  JOIN graph g ON r.source_id = g.entity_id
  WHERE g.depth < maxDepth
    AND NOT (r.target_id = ANY(g.path)) -- cycle prevention
)
SELECT DISTINCT ON (g.entity_id)
  g.*, e.name, e.entity_type, e.uid
FROM graph g
JOIN entity e ON e.id = g.entity_id
WHERE e.merged_with IS NULL
ORDER BY g.entity_id, g.depth ASC
Three Traversal Actions
Exposed via the traverse_graph MCP tool and REST API:

neighbors

Direct 1-hop connections. Fast. Shows immediate relationships like "authored document X", "covers topic Y".

path

Shortest path between two entities. Up to 4 hops. Returns ordered entity chain + relation types at each step.

related

All reachable entities within N hops. BFS expansion. Returns entities with depth, relation type, and mention counts.

Database Configuration

cortex-gen (read/write)

  • Own PostgreSQL + pgvector via Docker
  • Port 5434, pgvector extension enabled
  • 9 tables, 11 migrations (.cjs)
  • HNSW indexes on chunk, fact, entity embeddings
  • Raw SQL for pgvector ops, Knex builder for CRUD
  • Namespace-scoped access across all tables

Key Design Decisions

  • Single DB for everything (no external dependencies)
  • Content hash dedup prevents re-processing unchanged docs
  • Soft deletes: contradicted/superseded facts preserved
  • merged_with FK chain for entity dedup history
  • JSONB for flexible config and entity_types
  • camelCase mappers in Knex for JS conventions
